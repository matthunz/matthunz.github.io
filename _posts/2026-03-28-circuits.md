_Circuits ❤️ Category theory_

Digital circuits have a suprisingly beautiful model in category theory that are easy to understand but powerful enough to prove properties about real-world systems.

By modeling a circuit with category theory we get a strict guarantee that it is _correct by construction_ by proving it adheres to the axioms of categories.

We can define a circuit as a [_symmetric monoidal category_](https://leanprover-community.github.io/mathlib4_docs/Mathlib/CategoryTheory/Monoidal/Braided/Basic.html#CategoryTheory.SymmetricCategory) with _objects_ as numbers of wires and [_morphisms_](https://en.wikipedia.org/wiki/Morphism) as functions between groupings of wires[^1]. By using four-value logic[^2], short-circuits and disconnections in wires can also be modelled.

Well anyways that's enough jargon and hooplah about math stuff, let's actually make something! Let's write out some lean to prove things about the circuit of an **adder** that performs addition of binary numbers.

## Circuits in lean

Let's first understand how circuits can be used with the notation from category theory. First we'll examine the primitive gates used to operate on four-value logic, often referred to as _Belnap_ gates[^2].

```
abbrev and : (ofNat 2 : C) ⟶ 1

abbrev or : (ofNat 2 : C) ⟶ 1

abbrev not : (ofNat 1 : C) ⟶ 1
```

> We use Lean's `OfNat.ofNat` to tell the elaborator these numbers are actually objects in the category of circuits.

We'll also use a primitve `fork` morphism that splits a wire to help connect our circuits:

```
abbrev fork :  (ofNat 1 : C) ⟶ 2
```

## Proving theorems about a digital circuit

One way to prove properties about a circuit is by using what are known as denotational semantics. These semantics will _denote_ the behaviour of circuits from their wiring to mathematical notation, in our case Lean4.

We'll focus on _combinational circuits_ which are simple because they don't carry state, meaning that they can be _denoted_ as ordinary mathematical/Lean functions.

> Specifically, to model real-world circuitry, they are [Subtype](https://leanprover-community.github.io/mathlib4_docs/Init/Prelude.html#Subtype)s containing [Monotone](https://leanprover-community.github.io/mathlib4_docs/Mathlib/Order/Monotone/Defs.html#Monotone) functions.

> Sequential circuits can be modeled as infinite causal stream functions [^1], other complex real-world circuits are asynchronous and modeling them is an open area of research, if you figure it out HMU!

### NOT

![NOT schematic]({{ site.baseurl }}/assets/not.svg)

Let's start by proving the denotational semantics of the `not` gate.

Essentially, we want to say that `not` follows this truth table:

```text
0 => 1
1 => 0
```

As a theorem we could state _for any bit `x` a NOT gate returns `!x`_, where `!` is the NOT operator, which we can write in Lean as the following:

```
theorem not_def (x : Bool) : not.val #v[↑x] = #v[↑(!x)]
```

> `not_def`, short for the definition of `not`, takes some bit, or `Bool`, `x` and returns a proposition that inputting `x` into `not` (`not.val #v[↑x]`) is equivalent to `!x` (`#v[↑(!x)]`).

> `.val` is used to access the _value_ of the [Subtype](https://leanprover-community.github.io/mathlib4_docs/Init/Prelude.html#Subtype)
> `↑` is the coercion operator in Lean, we use it here to tell the elaborator to coerce the `Bool` bits to `BelnapLevel`s (an implementation of the 4-value logic mentioned before.)

> `#v[...]` is syntax to create a `Vector`, which is used here to represent groups of wires

We can prove this theorem by going through all cases of `x` and using reflection:

```
theorem not_def (x : Bool) : not.val #v[↑x] = #v[↑(!x)] := by cases x <;> rfl
```

### XOR

![XOR schematic]({{ site.baseurl }}/assets/xor.svg)

First, let's define some helper circuits to fork wires twice and copy pairs of bits:

```
def fork₂ : (ofNat 2 : CombinationalCircuit) ⟶ 4 := fork ⊗ₘ fork
```

> `⊗ₘ` is the tensor morphism notation from [`mathlib`](https://github.com/leanprover-community/mathlib4)

```
def copy : (ofNat 2 : CombinationalCircuit) ⟶ 4 := fork₂ ≫ (1 ◁ ((β_ 1 1).hom ▷ 1))
```

> `≫` is category composition operator, which in this case is isomorphic to function composition.

> `◁` and `▷` are the _whiskering_ operators for monoidal categories, these are used here to align the braiding with fork₂'s outputs

> `β_` is the notation for _braiding_ which we use here to cross the inner wires in the circuit

Now we can use `copy` with `and`, `or`, and `not` gates to define `xor`:

```
def xor : (ofNat 2 : CombinationalCircuit) ⟶ 1 := copy ≫ (and ⊗ₘ or) ≫ (not ⊗ₘ 𝟙 1) ≫ and
```

We can prove the semantics of `xor` in a similar way by going through all cases of both `x` and `y`:

```
theorem xor_def (x y : Bool) : xor.val #v[↑x, ↑y] = #v[↑((x && !y) || (!x && y))] := by
  cases x <;> cases y <;> rfl
```

### Half-adder

![Half-adder schematic]({{ site.baseurl }}/assets/half-adder.svg)

An **adder** is comprised of two `halfAdders`, so we'll have to define that first.

```
def halfAdder : (ofNat 2 : CombinationalCircuit) ⟶ 2 := copy ≫ (xor ⊗ₘ and)
```

```
theorem halfAdder_def
    (x y : Bool) :
    halfAdder.val #v[↑x, ↑y] = #v[↑((x && !y) || (!x && y)), ↑(x && y)] := by
  cases x <;> cases y <;> rfl
```

### Adder

![Adder schematic]({{ site.baseurl }}/assets/adder.svg)

Now we're finally ready for the full `adder`, which will use all our previous circuits in the form of two `halfAdders` and an `or` gate.

```
def adder : (ofNat 3 : CombinationalCircuit) ⟶ 2 :=
  (halfAdder ⊗ₘ 𝟙 1) ≫
  (1 ◁ (β_ 1 1).hom) ⊗≫
  (halfAdder ⊗ₘ 𝟙 1) ≫
  (𝟙 1 ⊗ₘ or)
```

```
theorem adder_def (x y z : Bool) :
    adder.val #v[↑x, ↑y, ↑z] =
      #v[↑((((x && !y) || (!x && y)) && !z) || (!(((x && !y) || (!x && y))) && z)),
         ↑((x && y) || (((x && !y) || (!x && y)) && z))] := by
  cases x <;> cases y <;> cases z <;> rfl
```
